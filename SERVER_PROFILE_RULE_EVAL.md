## Issues encountered

### 1. `PolicyBucket` create behavior is misleading and validation is too weak

The highest-signal issue is that `server.profile.create` accepted a `PolicyBucket` containing policies that do not look valid for a standalone server profile:

- `UnifiedEdgeFirmware` reports `TargetPlatform: UnifiedEdgeServer`
- `UnifiedEdgeNetworkConfig`, `UnifiedEdgeNtp`, `UnifiedEdgeSyslog`, and `UnifiedEdgeThermal` were already attached to `fabric.SwitchProfileTemplate` objects
- the created profile itself is `TargetPlatform: Standalone`

The create response showed those policies in `PolicyBucket`, but a fresh read of the persisted profile kept only four same-org custom policies. In other words, the initial response implied success, but the stored object silently dropped most of the requested attachments.

After switching to same-org custom policies, the profile successfully persisted twelve policies. The current rule layer appears to validate only request shape, not policy/profile compatibility or durable persistence outcomes.

Suggested rule improvements:
- restrict `server.Profile.PolicyBucket` to policy types explicitly supported by server profiles
- validate target-platform compatibility between profile and policy
- reject policy attachments where the policy is clearly scoped to a different profile family
- treat silent policy drops as validation failures instead of returning a misleading successful create payload

### 2. Resource naming is inconsistent for BIOS

The catalog key is `bio.policy`, but the schema/object type is `bios.Policy`.

Impact:
- users have to guess whether the resource name follows the REST path, SDK method, or object type
- prompt examples are harder to generalize across resources

Suggested rule/catalog improvement:
- normalize the exposed resource key or add aliases so `bios.policy` resolves cleanly

### 3. Timezone enum rejects `Etc/UTC`

Offline validation for `ntp.policy.create` rejected `Timezone: "Etc/UTC"` but accepted `Timezone: "UTC"`.

Impact:
- `Etc/UTC` is a common IANA timezone name and is likely to be tried by users or generated by tooling

Suggested improvement:
- allow `Etc/UTC`, or map it to `UTC` during validation

### 4. Offline validation misses backend-required constraints

Several payloads passed offline validation but failed against the live backend:

- `access.policy.create` passed offline, but the backend rejected it with `invalid_in_band_vlan_id` because `InbandVlan` effectively resolved to `0` and the backend requires `4-4093`
- `syslog.policy.create` passed offline, but the backend rejected it because `LocalClients` must contain at least one element

Impact:
- the current rule layer can report a payload as valid even when it is guaranteed to fail live
- users do not get actionable guidance until after a persistent API attempt

Suggested improvement:
- promote these backend constraints into the semantic rules layer
- do not rely only on shape/schema validation for policy resources with known server-side defaults

### 5. `sol.policy` enum validation is internally inconsistent

Offline validation rejected `BaudRate: 115200` even though the returned allowed enum list also included `115200`.

Impact:
- the validator response is self-contradictory
- users cannot tell whether the payload or the validator is wrong

Suggested improvement:
- fix the enum comparison for `sol.Policy.BaudRate`
- add a regression test for the exact `115200` case

### 6. Catalog coverage makes “complete” profile work harder than it should

During discovery, some resources that would help with full server-profile authoring were not exposed in the expected way:

- `compute.physical` was not available from the resource catalog for server assignment lookup
- `power.policy` was not present in the catalog search result even though it is a common server-profile concept

Impact:
- building a richer profile requires more trial-and-error than necessary
- the user has less help discovering what can be attached or assigned

Suggested improvement:
- audit server-profile-adjacent resource coverage in the search catalog
- ensure common server-related resources are discoverable under their expected names

### 7. System-defined and cross-scope policies are easy to attach without guidance

The profile accepted a mix of:

- custom policies in the `default` organization
- `SystemDefault` policies from the shared Cisco system organization
- catalog-defined `UnifiedEdge*` policies with existing switch-template associations

This may be allowed by the backend, but the rules layer currently gives no help distinguishing:

- safe baseline reuse
- cross-org shared system policies
- semantically incompatible policy reuse

Suggested improvement:
- add warnings or validation messages when attaching policies outside the profile's organization
- at minimum, surface whether the policy is system-defined or already associated with another profile family

## Notes

I intentionally avoided LAN, SAN, UUID pool, and server-assignment configuration in this run because those require additional dependent objects and would have mixed basic policy-bucket validation findings with pool/identity prerequisites.
